EKS Monitoring Setup Guide

Follow these steps to connect a new EKS cluster.

Step 1: Check for an Existing Node Group

Before deploying the monitoring agent, ensure your EKS cluster has a node group where the agent can run. You can check for existing node groups with the following command:

eksctl get nodegroup --cluster=your-cluster-name --region=your-region

Replace:
• your-cluster-name - Your actual EKS cluster name
• your-region - AWS region where your cluster is deployed (e.g., ap-south-1, us-east-1)

If no node group is found, you will need to create one using eksctl or AWS Console.

Step 2: Attach Required IAM Policy

The worker nodes in your cluster need permission to send metrics to CloudWatch. Attach the CloudWatchAgentServerPolicy to your node instance role using the AWS CLI:

aws iam attach-role-policy \
--policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy \
--role-name your-node-instance-role-name

Replace:
• your-node-instance-role-name - Your EKS node instance role name (typically starts with "eksctl-")

You can find your node instance role name in the AWS console under IAM > Roles or by checking your node group configuration.

Step 3: Deploy the CloudWatch Agent and Fluent Bit

Next, download and deploy the necessary monitoring agents to your cluster. This is done by applying a pre-configured YAML file from AWS.

First, set your cluster name and region as environment variables:

ClusterName='your-cluster-name'
RegionName='your-region'
HttpServer='On'
HttpPort='2020'
ReadHead='Off'
ReadTail='On'

Replace:
• your-cluster-name - Your actual EKS cluster name
• your-region - AWS region where your cluster is deployed

Then, download, configure, and apply the manifest (clean up any old file first):

# Clean up any old file
rm -f cwagent-fluent-bit-quickstart.yaml

# Download the latest manifest
curl -O https://raw.githubusercontent.com/aws-samples/amazon-cloudwatch-container-insights/latest/k8s-deployment-manifest-templates/deployment-mode/daemonset/container-insights-monitoring/quickstart/cwagent-fluent-bit-quickstart.yaml

# Apply substitutions
sed -i -e "s/{{cluster_name}}/$ClusterName/g" -e "s/{{region_name}}/$RegionName/g" -e "s/{{http_server_toggle}}/\"$HttpServer\"/g" -e "s/{{http_server_port}}/\"$HttpPort\"/g" -e "s/{{read_from_head}}/\"$ReadHead\"/g" -e "s/{{read_from_tail}}/\"$ReadTail\"/g" cwagent-fluent-bit-quickstart.yaml

# Apply the configured manifest
kubectl apply -f cwagent-fluent-bit-quickstart.yaml

Step 4: Patch the Agent Configuration

After deploying the agents, you need to patch the cwagentconfig ConfigMap to ensure the correct cluster name is being used. This is a crucial step for the monitoring data to appear in the correct CloudWatch Log Group.

Execute the following kubectl patch command:

kubectl patch configmap cwagentconfig -n amazon-cloudwatch -p '{"data":{"cwagentconfig.json":"{\n  \"agent\": {\n    \"region\": \"your-region\"\n  },\n  \"logs\": {\n    \"metrics_collected\": {\n      \"kubernetes\": {\n        \"cluster_name\": \"your-cluster-name\",\n        \"metrics_collection_interval\": 60\n      }\n    },\n    \"force_flush_interval\": 5\n  }\n}"}}'

Replace:
• your-region - AWS region where your cluster is deployed
• your-cluster-name - Your actual EKS cluster name

Step 5: Restart the DaemonSet

Finally, restart the cloudwatch-agent DaemonSet to apply the configuration changes and the new IAM role permissions:

kubectl rollout restart daemonset cloudwatch-agent -n amazon-cloudwatch

After completing these steps, it may take a few minutes for the monitoring data to start appearing in your CloudWatch console and on the XamOps dashboard.

Note: Make sure your kubectl context is configured to point to your EKS cluster before running these commands.
